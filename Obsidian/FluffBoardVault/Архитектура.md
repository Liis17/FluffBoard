# Архитектура

## Tech Stack

| Область | Технологии | Роль |
|---------|------------|------|
| Frontend | React 19, React DOM, Vite 8, JavaScript | Вход и Kanban-интерфейс |
| Backend | ASP.NET Core minimal API, .NET 10 | Cookie-сессии, API и GitHub-интеграция |
| Локальные данные | SQLite (`Microsoft.Data.Sqlite`) | Кеш пользователей доски и связей идентичностей |
| Внешние данные | GitHub REST API | Источник и место записи задач |
| Качество | Oxlint | Линтинг frontend-кода |

## Компоненты

- [[backend/API]] — работает на `http://localhost:5279`, загружает `.env`, хранит пользователей локально и публикует защищённое API доски.
- [[backend/GitHubClient]] — от имени сервисного GitHub-аккаунта читает и записывает issues одной репозитории.
- [[frontend/Приложение]] — Vite-клиент, обычно на `http://localhost:5173`; проксирует `/api` и хранит только cookie-сессию, не токен GitHub.
- `CONTEXT.md` — словарь доменных терминов: пользователь доски, GitHub assignee, Telegram identity и задача.

## Сквозной поток: вход и изменение задачи

1. Backend загружает `.env`, валидирует owner/repository и список пользователей, затем синхронизирует пользователей в SQLite.
2. Пользователь отправляет логин и пароль в `POST /api/auth/login`; backend сравнивает пароль с PBKDF2-хешем и создаёт cookie-сессию.
3. `App` запрашивает `/api/board/issues`, `/labels` и `/users` через Vite proxy.
4. Backend передаёт запрос `GitHubClient`; тот читает свежие данные GitHub и исключает pull request'ы.
5. При сохранении редактор отправляет POST или PUT в backend. `GitHubClient` создаёт/обновляет issue GitHub и возвращает новую версию.
6. Клиент перезагружает доску из GitHub и раскладывает задачи по labels `todo`, `in-progress`, `done`; закрытая GitHub issue также показывается в «Готово».

## Соглашения

- Одна инсталляция FluffBoard обслуживает одну репозиторию, заданную в `Board__Repository__Owner` и `Board__Repository__Name`.
- Владелец `GitHub__Token` — отдельный сервисный аккаунт с минимальным permission **Issues: Read and write**. Браузер не получает этот токен.
- Локальный пользователь может иметь `GitHubLogin` и `TelegramId`: это одна личность в разных системах. Telegram-авторизация и уведомления остаются будущим расширением.
- Workflow — GitHub labels, а не отдельное состояние в SQLite. Label `done` не закрывает issue: закрытие остаётся отдельным состоянием GitHub.
