# FluffBoard

FluffBoard — минимальное full-stack приложение с React 19/Vite frontend и ASP.NET Core/.NET 10 backend. Frontend запрашивает приветствие у `GET /api/hello`.

## База знаний — Obsidian

Vault находится в `Obsidian/FluffBoardVault/`.

| Путь | Содержимое |
|------|------------|
| `Obsidian/FluffBoardVault/Index.md` | Навигация по проекту |
| `Obsidian/FluffBoardVault/Архитектура.md` | Стек, поток запроса и соглашения |
| `Obsidian/FluffBoardVault/backend/API.md` | Backend minimal API и конфигурация |
| `Obsidian/FluffBoardVault/frontend/Приложение.md` | React-клиент и его загрузочный поток |

## Правила обновления Obsidian

- Изменилась функциональность файла или метода — обнови заметку соответствующего компонента. Устаревший метод помечай `~~methodName~~ (удалён: [дата])`, новый добавляй в таблицу методов.
- Добавлен файл или компонент — создай либо дополни заметку в реальном домене (`backend` или `frontend`) и добавь ссылку в `Index.md`.
- Изменилась общая архитектура, зависимости, порты или конфигурация — обнови `Архитектура.md`.
- Changelog не веди: история изменений хранится в Git. Wikilinks используй в формате `[[Файл]]` или `[[Домен/Файл]]`.

## Правила чтения файлов проекта

- Для исследования большого репозитория сначала собирай структуру и домены через Explore с моделью haiku; для этого небольшого проекта достаточно прямого чтения ключевых файлов.
- Перед созданием заметки проверяй, нет ли существующей через поиск по имени и содержимому vault.
- Для правок кода и заметок Obsidian читай конкретные целевые файлы напрямую; перед переименованием заметки проверяй backlinks.

## Поведенческие правила

Правила для снижения типичных ошибок при написании кода.

**Компромисс:** эти правила смещают баланс в сторону осторожности, а не скорости. Для тривиальных задач — действуй по ситуации.

### 1. Думай перед кодом

**Не предполагай. Не скрывай сомнения. Показывай компромиссы.**

Перед реализацией:

- Явно формулируй свои допущения. Если не уверен — спроси.
- Если возможны несколько трактовок — покажи их, не выбирай молча.
- Если есть более простой подход — скажи об этом. Возражай, когда это оправдано.
- Если что-то неясно — остановись. Назови, что именно сбивает с толку. Спроси.

### 2. Простота прежде всего

**Минимум кода, решающего задачу. Ничего спекулятивного.**

- Никаких функций сверх запрошенного.
- Никаких абстракций для одноразового кода.
- Никакой «гибкости» или «настраиваемости», которую не просили.
- Никакой обработки ошибок для невозможных сценариев.
- Если написал 200 строк, а можно 50 — перепиши.

Спроси себя: «Сказал бы senior-инженер, что это переусложнено?» Если да — упрости.

### 3. Хирургические правки

**Трогай только то, что необходимо. Убирай только свой беспорядок.**

При редактировании существующего кода:

- Не «улучшай» соседний код, комментарии или форматирование.
- Не рефактори то, что не сломано.
- Соблюдай существующий стиль, даже если сделал бы иначе.
- Если заметил несвязанный мёртвый код — упомяни, но не удаляй.

Когда твои изменения создают «сирот»:

- Удаляй импорты/переменные/функции, которые стали неиспользуемыми ИЗ-ЗА твоих изменений.
- Не удаляй ранее существовавший мёртвый код без запроса.

Тест: каждая изменённая строка должна напрямую следовать из запроса пользователя.

### 4. Выполнение, ориентированное на цель

**Определи критерии успеха. Повторяй до проверки.**

Превращай задачи в проверяемые цели:

- «Добавь валидацию» → «Напиши тесты на невалидный ввод, затем сделай, чтобы они проходили»
- «Исправь баг» → «Напиши тест, воспроизводящий баг, затем сделай, чтобы он проходил»
- «Отрефактори X» → «Убедись, что тесты проходят до и после»

Для многошаговых задач кратко изложи план:

```
1. [Шаг] → проверка: [контроль]
2. [Шаг] → проверка: [контроль]
3. [Шаг] → проверка: [контроль]
```

Сильные критерии успеха позволяют работать автономно. Слабые («просто заставь работать») требуют постоянных уточнений.

**Эти правила работают, если:** в диффах меньше лишних изменений, меньше переписываний из-за переусложнения, а уточняющие вопросы возникают до реализации, а не после ошибок.
